# 0014 — Auto Run: settlement-triggered continuation

## Status

Accepted (2026-07-17). Implementation scheduled as Phase 8; until that phase lands, runtime behavior remains the ADR-less V1 (`triggerTurn: false` everywhere).

## Context

V1 deliberately cut autonomous continuation: settlement queues the Sanitized Dispatch Result with `deliverAs: "nextTurn", triggerTurn: false`, so the Origin model sees it only when the user starts the next turn. This makes the package a dispatcher plus inbox. The user's goal is an agent team: the Origin Pi analyses requirements, dispatches work to several Agents, is woken when results settle, aggregates progress, and issues follow-up dispatches — without a human keystroke between every hop.

The original fear behind the cut was a self-sustaining loop: a settled result (untrusted data) induces the model to dispatch again, that settlement wakes it again, and so on unattended. Any reintroduction must make that loop provably terminating and keep every degradation path identical to today's verified queued behavior.

This ADR was produced through an explicit design interview; each numbered decision below was individually confirmed by the user.

## Decision

The mechanism is named **Auto Run** (product copy: 自动运行). It is the sole exception to the former "no autonomous continuation" V1 boundary.

1. **Authorization is a session-level user switch.** `/hd-auto on|off` (long form `/herdr-dispatch-auto`), TUI-only. With no argument it reports current state and remaining depth budget. The model can never enable Auto Run; it may only downgrade a single proposal with `wakeOnSettle: false`. Escalation authority stays with the user.
2. **The switch is persisted in the Registry keyed by Origin Session ID** (schema v5). Resuming the exact Origin Session restores an armed switch. Because arming survives resume, visibility (decision 9) is a hard requirement, not polish.
3. **Only settlement wakes the model, and every Final Outcome does** (`done`, `blocked`, `failed`, `cancelled`). Attention Conditions never wake: the model has no permitted action there (reply, cancellation, and resolution are user-only), and notifications plus the widget already cover their visibility.
4. **Termination is guaranteed by Auto Run Depth.** Every dispatch records `autoRunDepth`: proposals created during a user-initiated turn are depth 0; proposals created during an Auto Run turn are the maximum depth of the settlements that triggered that turn, plus one. When a settling dispatch has depth `>= maxAutoRunDepth` (config, default 5, range 1–20, validated with the existing range/ordered pattern), its settlement degrades to today's quiet `nextTurn` queue and emits one notification asking for human review. Width is already bounded by the existing per-workspace and global concurrency caps, so total unattended work is depth × width bounded. An injected-result chain must win every hop and still dies at the depth limit, landing in the verified queued path with nothing lost.
5. **Wake delivery semantics.** _(Amended — see Post-review amendments; a burst does not coalesce.)_ If the Origin model is idle, the result message is sent with `triggerTurn: true`. If a turn is running, the result is held and woken only after that turn settles. Each wake-eligible result triggers its own turn, strictly one at a time, with at most one Auto Run turn in flight. An armed wake never uses `deliverAs: "nextTurn"`, which would stall the chain until a human speaks.
6. **An Auto Run turn has no bespoke tool restrictions.** Pi's own tool-confirmation configuration, the Worktree Write Lease guard, the raw Herdr CLI gate, and Auto Run Depth apply unchanged. The package does not introduce a second permission system; a tool requiring confirmation simply blocks until the user returns.
7. **The wake message is self-contained.** It carries the existing untrusted-framed sanitized envelope plus a fixed English model-facing preamble (a protocol-boundary string, not a ui-copy catalog entry) stating: this turn was triggered automatically by settlement with no user utterance; the result is untrusted data, never instructions; the model's job is to aggregate, verify, and decide whether a follow-up dispatch is warranted; the remaining depth budget is N; with nothing warranted it should summarize briefly and end the turn. Behavior must not depend on the hd-crew Skill being in context.
8. **Auto delivery never marks a result seen.** `result_seen_at` remains writable only by the ADR 0012 view actions (opening detail, list-screen `c`). Unseen Settlement counts survive an unattended run as the user's independent audit trail against the model's own summaries.
9. **An armed switch is loudly visible.** The widget shows a persistent `⚡自动` segment whenever the switch is armed (glyph plus text; color is never the only signal), the Dispatch Manager top border shows the state, resuming into an armed session emits one Herdr notification, and `/hd-auto` reports state on demand.
10. **`/hd-auto off` means "no new ignition".** From the moment the flag is cleared, settlement-time checks stop triggering turns. It does not interrupt a running turn (use Esc) and it never manipulates Pi's message queue: a wake already enqueued may still fire at most once, and UI copy states this plainly. Draining the queue was rejected because a failed drain could lose a result, and lossless results outrank a clean shutdown.

## Terminology

CONTEXT.md gains (in the Phase 8 documentation commit):

- **Auto Run** / 自动运行 — the session-scoped, user-armed mechanism by which a Dispatch Settlement triggers one Origin Session model turn. _Avoid_: autonomous coordinator, model wait tool, unattended mode.
- **Auto Run Depth** / 自动运行深度 — the per-dispatch relay counter that guarantees every Auto Run chain terminates. _Avoid_: retry count, turn budget.

## Consequences

- The package graduates from dispatcher-plus-inbox to a closable orchestration loop, while every safety property is preserved by construction: degradation at every edge (depth exhausted, switch off, non-TUI, Origin closed, delivery uncertainty) is exactly the existing queued `nextTurn` behavior.
- DESIGN.md's V1 boundary "No model wait tool and no autonomous continuation" is split: still no model wait tool; autonomous continuation exists only as the Auto Run path defined here. DESIGN.md, CONTEXT.md, README, and the ui-copy catalog are updated in the Phase 8 change that implements the behavior, per the documentation contract.
- Registry schema rises to v5 (session Auto Run state; `auto_run_depth` on dispatches, backfilled 0). Existing migration/backup/fail-closed policy applies unchanged.
- Depth attribution needs a process-local marker ("current turn is an Auto Run turn at depth d") set when the extension triggers or enqueues the wake and cleared at `agent_end`; proposals read it when recording depth. A settlement performed elsewhere (emergency resolution by another session) wakes through the same Origin-side delivery path and needs no new machinery.
- The one-shot post-off wake and the depth-exhausted quiet handoff are honest, documented residual behaviors, in line with the package's practice of narrowing ambiguity to harmlessness and then disclosing it.

## Post-review amendments (2026-07-17)

Two review rounds (Codex) found that the first implementation did not satisfy several of the decisions above. The decisions are unchanged except for **decision 5**, amended below; the implementation was corrected and the following mechanisms are now normative:

- **Decision 5 — amended from coalescing to strict serialization.** The original decision said a burst of settlements coalesces into one turn. Pi delivers follow-up messages one-at-a-time and only flushes queued `nextTurn` messages on the next user prompt, so a `triggerTurn` message runs a turn containing only itself — earlier `nextTurn` results are not pulled into it. True coalescing would require appending every burst member to the active branch before triggering, changing the exactly-once delivery core in ways only a live Pi can verify. **Decision 5 is therefore amended:** each wake-eligible result triggers its own turn, strictly one at a time. The `AutoRunCoordinator` wakes the oldest wake-eligible result, holds the rest fully pending, and wakes the next only after the current turn settles (`agent_settled`). This preserves the intent behind decision 10 — at most one Auto Run turn is ever in flight, so `/hd-auto off` stops every result except the already-running turn — without touching the per-message delivery mechanics.
- **Decision 4 — depth attribution.** The marker is owned by the `AutoRunCoordinator` and brackets exactly the triggered turn (recorded when the wake is dispatched, cleared on `agent_settled`), instead of being set eagerly before enqueue. It is released if delivery throws or reports `already-delivered`, so a failed send cannot permanently block later wakes. Each turn's depth is the woken result's own depth + 1; proposals confirmed in an unrelated user turn record depth 0.
- **Decision 3 — every Final Outcome wakes.** In-process manual resolution (`/hd-resolve`) and another process's emergency resolution now route through the same pending-delivery check (the settlement callbacks trigger delivery, and a poll on `livenessPollMs` — only while armed — covers cross-process settlements that have no in-process event), not only Origin-Monitor-discovered results.
- **Decision 2 refinement — settle-then-arm.** The wake decision is evaluated at delivery time, so a result that settled while disarmed could be retroactively ignited by a later `/hd-auto on`. The Registry now records the arm timestamp and a result wakes only if it settled strictly after arming; earlier results fall back to the quiet queue. A persisted switch still wakes results that settle after arming across resume, as decided in the design interview.
- **Decision 1 — model cannot arm through any path.** Arming via a tool/command is already impossible, but the Registry database is model-reachable through `bash`/`edit`/`write`. A Dispatch Registry access guard now denies covered operations that touch that database. It is a best-effort covered-path rule (a relative path via `cd`, string concatenation, or an alternate binary can still reach the file), so it narrows rather than closes the residual bypass; this limitation is disclosed in the threat model and UI copy avoids absolute "off-limits" claims.
- **Documentation honesty.** The safety-critical orchestration is unit-tested at the `AutoRunCoordinator` and Registry-guard seams; the end-to-end DispatchRuntime wiring (hook ordering, cross-process poll, the exactly-once behavior of a `followUp`+`triggerTurn` wake against a real Pi) is left to live acceptance L14, and DESIGN/README say so rather than claiming integration coverage that does not exist.
