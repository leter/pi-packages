# Re-read streamed results through the startup window

Herdr's `pane.output_matched` subscription matches the Dispatch Correlation ID as soon as the outbound prompt is rendered. That first match usually contains the Result Envelope contract template, not the target's later result, and the subscription may not emit another useful event for JSON continuation rows. A three-read sequence ending after about 1.4 seconds passed mocked tests but failed repeatedly against real Codex, whose valid hard-wrapped Result Envelope appeared roughly six seconds later. The Registry remained `active` with false `malformed-result` attention even though the complete valid result was visible in the bounded tail.

After a matching output event, the Origin Monitor now performs exponentially spaced 200-line re-reads bounded by the configured startup window. Incomplete matching JSON is not marked malformed during that window. The exact outbound contract template (`done|blocked|failed|cancelled` plus `"..."`) is delivery evidence and is ignored as a result attempt. A complete valid envelope settles immediately; only evidence still malformed on the final bounded read adds `malformed-result`. Prefix-only rows may carry TUI decorations such as Grok's right-aligned timestamp; the scanner discards that decoration before joining the following hard-wrapped JSON. Transient read failures do not terminate later bounded attempts, while subscription state remains responsible for transport attention.

This does not add a wait tool, unbounded polling, continuing output surveillance, automatic resend, or an autonomous Origin model turn. It spends a small bounded number of reads to bridge TUI rendering latency already covered by the startup window.
