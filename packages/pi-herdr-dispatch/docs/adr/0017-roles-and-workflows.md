# 0017 ‚ÄĒ Roles and staged workflows for Board Tasks

## Status

Accepted (2026-07-17); design agreed in the ADR 0016 interview, direction re-confirmed by the user before implementation. Builds on the Task Board (ADR 0016). Implementation contract: [ROLES-WORKFLOWS-SPEC.md](../ROLES-WORKFLOWS-SPEC.md).

## Context

The Task Board gives unattended nights a durable queue, but routing knowledge still lives in each task's prose: who should implement, who reviews, what happens when review fails. The user wants a configurable team ‚ÄĒ write code / review / fix bugs / chores / research, plus a strongest-model consultant ‚ÄĒ and repeatable pipelines (dev work goes implement ‚Üí review; research is single-stage) without retyping that choreography into every task. The intelligence split stays as established: deterministic structure (stages, thresholds) belongs in configuration, judgment (which eligible pane, when to consult) belongs in the hd-crew Skill executed by the Origin model, and the extension itself never schedules.

## Decision

1. **One optional config file, `~/.config/pi-herdr-dispatch/team.json`, with built-in defaults.** Missing file = built-in roles and workflows; a present file overrides by key. Invalid content fails closed for board dispatching exactly like invalid `config.json` does for its surface, with one clear notification.
2. **Seven built-in roles:** `coder` (ŚľÄŚŹĎ, write), `reviewer` (ŤĮĄŚģ°, non-mutating), `bugfix` (šŅģbug, write), `chore` (śĚāśīĽ, write), `researcher` (ŤĶĄśĖô, non-mutating), `advisor` (ť°ĺťóģ, non-mutating), `oracle` (ÁĽąŚģ°, non-mutating). Each role: `label` (zh-CN product copy), default `mode`, and a `brief` ‚ÄĒ an English role preamble prepended to the outbound task text as part of the immutable payload. Briefs are advisory like every target constraint.
3. **Role-to-Agent binding is by pane-name match at routing time.** A Board Task may carry an optional role. The hd-crew Skill routes it to an Eligible Agent whose pane name contains the role key (e.g. panes named `coder-1`, `reviewer`, `oracle`), falling back to any Eligible Agent with a plain disclosure when no named pane fits. The user prepares role capacity by naming panes when setting up ŤĹ¶šĹć; the extension still never creates panes or worktrees.
4. **Named workflows with linear stages.** Built-ins: `dev` = `["coder", "reviewer"]`, `research` = `["researcher"]`, `quick` = `["chore"]`. A task carries an optional workflow (defaulted from its role: coder‚Üídev, researcher‚Üíresearch, chore‚Üíquick, none‚Üísingle-stage). The Registry stores the task's current stage index; a stage's dispatch settling advances or repeats the stage per decisions 5‚Äď6, and only the final stage's settlement moves the task to `review` for human acceptance. Stage advancement is recorded transactionally with settlement, but the *next dispatch* is still model-routed in a wake or user turn ‚ÄĒ no autonomous scheduler.
5. **Review stages emit a structured verdict.** A `reviewer`-stage outbound contract adds one optional Result Envelope field: `verdict: "pass" | "needs-rework"`. `pass` advances the stage; `needs-rework` sends the task back to its implement stage carrying the reviewer's summary as rework feedback (same framed block as a user return, marked untrusted). A missing or malformed verdict never guesses: the task parks in `review` for the human, disclosed as ŤĮĄŚģ°śú™ÁĽôÁĽďŤģļ.
6. **Rework budget and escalation are per task and separate from Auto Run Depth.** `maxReworkCycles` (default 2) counts needs-rework loops per task. The workflow may declare an escalation chain (default for `dev`: after 2 cycles hand the implement stage to `bugfix`, after 4 to `oracle`); each handoff carries the accumulated review feedback as untrusted data. When the budget and chain are exhausted, the task parks in `review` labelled ŤĮĄŚģ°śú™ŤŅá with its history. Every stage dispatch remains a task-bound dispatch: depth 0, Run Quota consumed while armed, full typed-path validation.
7. **Oracle discipline lives in the Skill, not in enforcement.** hd-crew gains: `advisor`/`oracle` panes are never used for ordinary stages; `oracle` receives only exhausted escalations and verdict arbitration; which model an oracle pane runs is entirely the user's pane setup. The extension adds no new model tools; `herdr_task_draft` gains optional `role` and `workflow` fields (validated against the loaded catalog), and /hd-task offers the same as wizard selects.

## Consequences

- The overnight loop gains division of labor without new authority: every stage dispatch is still user-pre-authorized via task approval, quota-counted, depth-0, and human-accepted at the end. Escalation widens who works, never what is permitted.
- Registry schema rises to v7 (task role/workflow/stage/rework counters; verdict on results). Existing migration/backup policy applies.
- A malicious or confused reviewer can emit a false verdict; the mitigation is the human acceptance gate plus the untrusted framing on all relayed feedback, and `verdict` never releases anything ‚ÄĒ it only chooses the next advisory hop.
- README, DESIGN.md, CONTEXT.md, ui-copy, and hd-crew SKILL.md update in the implementing change. Live acceptance L17 must cover: role routing by pane name with fallback disclosure, a dev-workflow pass verdict advancing to human review, a needs-rework loop returning to coder with framed feedback, escalation to bugfix after the configured cycles, budget exhaustion parking as ŤĮĄŚģ°śú™ŤŅá, and a missing verdict parking for the human.
